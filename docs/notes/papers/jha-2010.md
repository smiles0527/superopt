# Jha, Gulwani, Seshia, Tiwari — Oracle-Guided Component-Based Program Synthesis (ICSE 2010)

> Read twice before starting Phase 4.

## Citation
Jha, S., Gulwani, S., Seshia, S. A., & Tiwari, A. (2010).
*Oracle-Guided Component-Based Program Synthesis.*
ICSE 2010.

## Why it matters here

This is the encoding behind Phase 4. The core move: don't enumerate programs —
fix a *multiset of components* and let the SMT solver pick the wiring
(a.k.a. "location variables").

## The three queries to internalize

1. **Synthesis query.** Given a finite set of input/output examples, find a
   wiring of the components that satisfies all of them. The unknowns are the
   location variables (and any constant slots).
2. **Verification query.** Given a candidate program, ask: ∃ input where it
   disagrees with the spec? UNSAT ⇒ done and correct. SAT ⇒ extract the
   counterexample, add it to the example set, re-synthesize.
3. **Well-formedness constraints.** Every line has exactly one definer; no
   cycles; each component used once. **This is the off-by-one silent-bug
   zone — derive these yourself before reading anyone else's code.**

## The free-constant trick

Brute-force enumeration over constants is hopeless at 32-bit. CEGIS treats each
constant slot as a free `BitVec` variable — the solver finds the constant as
part of the wiring. This is the single biggest reason CEGIS scales where
enumeration doesn't. _Write this up; it's a real insight worth a paragraph in
the report._
